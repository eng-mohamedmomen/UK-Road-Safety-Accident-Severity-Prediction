# UK Road Safety — Accident Severity Prediction
## Final Project Report: Challenges, Findings, and Honest Assessment

---

## 1. Project Objective

Predict whether a UK road accident (STATS19 dataset, 2005–2017, 2,046,228 records) results in a **Slight** injury versus a **Serious or Fatal** outcome, using accident conditions known before or at the time of the crash (road type, speed limit, weather, lighting, junction type, time of day, etc.).

The project criteria required the final model to achieve **Precision ≥ 0.3 AND Recall ≥ 0.3** simultaneously on a held-out test set.

---

## 2. Final Result

| Metric | Value | Required | Met? |
|---|---|---|---|
| Precision | 0.285 | ≥ 0.30 | ❌ (1.5 points short) |
| Recall | 0.300 | ≥ 0.30 | ✅ |
| F1 | 0.292 | — | — |
| Accuracy | 0.778 | — | — |

The best-performing model — a PyTorch neural network trained on the full 2,046,228-row dataset with `pos_weight` handling class imbalance — came within **1.5 percentage points of Precision** of meeting both criteria simultaneously. Every classical baseline (Logistic Regression, Decision Tree, XGBoost, Naive Bayes) plateaued well below this, between 0.20 and 0.23 Precision.

This report documents **why** this ceiling exists, what was tried, and what it reveals about the dataset itself.

---

## 3. The Core Problem: Severe Class Imbalance

The target variable `Severity` is split:
- **Slight: 84.7%** (1,733,597 accidents)
- **Serious/Fatal: 15.3%** (312,872 accidents)

This is a roughly **5.5:1 imbalance**. Standard classifiers trained on imbalanced data default to predicting the majority class almost everywhere, since doing so already achieves 84.7% accuracy with zero effort. Every technique applied in this project was, at its core, an attempt to force the model to pay attention to the rare 15.3% class without breaking its ability to correctly reject the 84.7% majority.

### Why this is hard mathematically

With a test set of 409,246 rows (62,527 Serious/Fatal, 346,719 Slight), hitting Precision ≥ 0.3 with Recall ≥ 0.3 requires:
- At least **18,758 True Positives** (30% of all real Serious/Fatal cases correctly caught)
- No more than **43,769 False Positives** (Slight accidents incorrectly flagged as Serious/Fatal)

That ratio — needing to misclassify fewer than 12.6% of the 346,719 majority-class rows while still catching 30% of the minority class — is a narrow target. Every False Positive directly erodes Precision because the majority class is so much larger than the minority class; even a small majority-class error rate produces a False Positive count that dwarfs the True Positive count.

---

## 4. Approaches Attempted (In Order)

### 4.1 Baseline models with `class_weight='balanced'`
Logistic Regression, Decision Tree, Random Forest, XGBoost, and Naive Bayes were trained with built-in class weighting. **Result: Precision capped around 0.16–0.23.** The weighting nudged recall up but never pushed precision past the 0.2 range — the models were still fundamentally guessing on borderline cases.

### 4.2 SMOTE oversampling
Synthetic Minority Over-sampling Technique was applied to balance the training data to 50/50 before retraining. **Result: marginal improvement, same ceiling.** Two structural problems emerged:
- SMOTE on the full dataset is computationally impossible — building the K-Nearest-Neighbor distance matrix across 313,072 minority rows would require an estimated **392 GB of RAM**, far beyond Google Colab's ~12–15 GB limit. SMOTE had to be restricted to small samples (200–500K rows), which reintroduced information loss.
- Synthetic minority points generated by SMOTE are *interpolations* between real minority examples. In a dataset where the minority class is defined by a complex combination of conditions (speed × road type × lighting × junction), synthetic points can land in regions that don't correspond to real-world accident patterns, teaching the model rules that don't generalize to the actual test set.

### 4.3 Probability calibration + threshold tuning
Raw model probabilities were recalibrated using `CalibratedClassifierCV` (isotonic regression), and the decision threshold was scanned exhaustively to find the operating point that maximized Precision subject to Recall ≥ 0.3. **Result: this was the single most effective lever** — it took the XGBoost+SMOTE pipeline from Precision 0.22 to Precision 0.25, and the final PyTorch model from an uncalibrated 0.22 to a calibrated 0.285. But it could not manufacture signal the model didn't already have; threshold tuning trades recall for precision along an existing curve, it cannot move the curve itself.

### 4.4 Feature selection — helped, then hurt
Three feature selection methods (Chi-Square filter, Random Forest embedded importance, RFE wrapper) were run and combined, reducing 41 encoded features down to 11 "agreed" features. This was initially intended to reduce noise, but **it removed exactly the granular signal needed to separate borderline cases** — features like `Latitude`/`Longitude` and detailed junction/road-class information that don't score highly in isolated statistical tests but matter in the specific combinations that define a serious accident. Reverting to the full 41-feature set improved results meaningfully.

### 4.5 Full-dataset PyTorch neural network (final approach)
Rather than sampling and oversampling, the final approach trained a 3-hidden-layer neural network (256→128→64→1, BatchNorm, Dropout, ReLU) on the **full 1,636,982-row training set** using mini-batch gradient descent (batch size 4096) on a Colab T4 GPU. Class imbalance was handled inside the loss function itself via `pos_weight` in `BCEWithLogitsLoss` — mathematically equivalent to oversampling the minority class 5.5×, but with zero additional memory cost and no synthetic data risk.

This was the best-performing approach by a clear margin (Precision 0.285 vs. 0.20–0.23 for all classical baselines), and it makes intuitive sense: it is the only approach that let the model see **all 313,072 real minority-class examples** during training, rather than a sample or a synthetic approximation of them.

## 5. Why a Precision/Recall Ceiling Exists in This Data

After exhausting model architecture, resampling, calibration, and feature engineering, the persistent ~0.28–0.30 ceiling points to a property of the **data itself**, not the modeling pipeline:

### 5.1 Overlapping feature distributions between classes
For a model to separate two classes well, the *combinations* of feature values that produce each class need to be reasonably distinct. In this dataset, accidents with **identical recorded conditions** — same road type, same speed limit, same weather, same junction detail, same lighting — appear in *both* the Slight and the Serious/Fatal class. This is expected and explainable: STATS19 records the *conditions surrounding* an accident, not the full causal chain (driver behavior, exact speed at impact, reaction time, vehicle type/safety rating, exact collision angle, etc.), all of which materially affect whether a crash is Slight or Fatal but none of which are captured in this dataset. Two accidents with the exact same road and weather conditions can have wildly different outcomes depending on factors STATS19 simply does not record.

This means a meaningful fraction of the "irreducible error" in this problem is not addressable through better modeling — it reflects a genuine information gap in the available features relative to the true causal drivers of severity.

### 5.2 Severity is partly a function of unmeasured factors
Real-world accident severity depends heavily on:
- Exact vehicle speed at the moment of impact (we only know the road's *speed limit*, not actual speed)
- Vehicle type, age, and safety rating (not in the dataset)
- Number and physical vulnerability of people involved (partially captured via `Number_of_Casualties`, which was excluded from modeling to avoid target leakage, since casualty count is itself a consequence of severity, not a predictor of it)
- Driver state (fatigue, alcohol, distraction — not in the dataset)

Without these, the model is working with a structurally weaker signal than the true generating process of accident severity.

### 5.3 Practical implication
The gap between our best result (Precision 0.285) and the criteria (0.300) is small in absolute terms but reflects a real and explainable data limitation rather than a modeling failure. Further gains would likely require either:
- Additional data sources merged in (vehicle registration data, weather severity indices, traffic volume data)
- Accepting a different precision/recall tradeoff point depending on the real-world cost of false negatives vs. false positives
- A fundamentally different framing of the target (e.g., predicting *risk score* rather than a hard binary classification)

---

## 6. What Was Learned

1. **Class imbalance is a data problem before it is a model problem.** No model architecture — from Logistic Regression to a full neural network — could overcome it through technique alone; the loss function's treatment of the imbalance (`pos_weight`) mattered more than the choice of algorithm.

2. **SMOTE has hard computational limits at scale** and risks teaching the model patterns that don't exist in real data when minority classes are large and complex.

3. **More features can outperform "cleaner" feature-selected sets** when the minority class is defined by feature combinations rather than individual strong predictors — feature selection methods that test features in isolation can discard genuinely useful interaction signal.

4. **Threshold tuning is a real and necessary final step**, not an afterthought — it was responsible for the largest single jump in precision in this project, but it has a ceiling defined by the model's underlying discriminative ability.

5. **A model evaluation gap (0.285 vs 0.300) can be a legitimate finding about data limitations**, not necessarily evidence of insufficient modeling effort. Documenting *why* a ceiling exists is as valuable a deliverable as hitting an arbitrary metric threshold.

---

## 7. Recommendations for Future Work

- Acquire supplementary data (vehicle specifications, exact recorded speeds where available, traffic density) to add genuinely new signal rather than re-deriving more from the existing feature set
- Explore reframing as an ordinal regression (Slight → Serious → Fatal) rather than a binary collapse, which may recover information lost in the binarization step
- Test cost-sensitive learning with an explicit, business-justified cost matrix rather than optimizing for a fixed 0.3/0.3 threshold, since real-world deployment would weigh false negatives (missed serious accidents) against false positives (unnecessary alerts) according to actual operational costs
- Consider ensemble stacking of the neural network with calibrated XGBoost, since the two models may make complementary errors
